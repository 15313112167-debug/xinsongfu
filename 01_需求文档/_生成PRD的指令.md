# 生成 PRD 的指令文档

> 把这份文档和 `PRD-模板.md` 一起放在项目根目录。当你需要让 Claude Code / Codex 根据原型图或产品资料生成需求文档时，参照本文档使用。

---

## 一、使用方式

### 场景 A：基于 Figma / Pencil 原型链接

在 Claude Code 输入框输入：

```
请基于以下信息生成产品需求文档：

【产品名称】：天算
【功能模块】：用户登录
【原型链接】：https://figma.com/file/xxx?node-id=1:234
【需求背景】：[用 2-3 句话描述背景和目标]
【目标用户】：[谁会用这个功能]

要求：
1. 严格按照 PRD-模板.md 的结构和章节组织内容
2. 通过 figma mcp（或 pencil mcp）读取原型，提取所有界面元素
3. 把元素填入「界面元素说明」表格，包含元素类型、默认值、校验规则
4. 业务流程表格逐步骤填写，区分用户动作和系统响应
5. 主动识别并补充异常流程和分支流程
6. 数据字典根据界面字段推断字段类型、必填、长度
7. 输出 .md 文件到 02-方案/ 目录，文件名格式：PRD-[功能名]-V1.0.md

use the figma mcp server
```

> 用 Pencil 原型时，把最后一行改成 `use the pencil mcp server`。

---

### 场景 B：基于原型截图

直接把截图拖到 Claude Code 对话框，附上指令：

```
请基于这张原型截图，按 PRD-模板.md 的结构生成需求文档。

【产品名称】：天算
【功能模块】：[功能名]
【需求背景】：[描述]

要求：
1. 仔细识别截图中的所有界面元素（输入框、按钮、文字、图标等）
2. 推断每个元素的类型、默认状态、可能的校验规则
3. 基于功能特征推断典型业务流程、异常和分支
4. 数据字典根据可见字段合理推断
5. 不确定的部分用「待补充」标注，不要凭空捏造
6. 输出到 02-方案/PRD-[功能名]-V1.0.md
```

---

### 场景 C：基于纯文字描述

```
请根据下面的产品介绍，按 PRD-模板.md 的结构生成需求文档：

[在这里粘贴你的产品介绍、用户故事、需求说明等]

要求：
1. 严格遵循模板结构，章节不缺失
2. 信息不足的部分用「待补充」标注
3. 主动识别需要补充的关键信息，在文档末尾列出 TODO
4. 输出到 02-方案/PRD-[功能名]-V1.0.md
```

## 二、对 AI 的核心要求

### 必须做到

1. **结构完整**：严格遵循 PRD-模板.md 的章节顺序，不省略也不擅自合并
2. **表格规范**：所有表格保持模板格式，列名、对齐方式一致
3. **流程闭环**：每个业务流程表格的最后一步必须是「流程结束」
4. **异常覆盖**：每个功能模块都要思考至少 1-2 个异常场景
5. **校验明确**：界面元素的校验规则要具体（如「必填、11 位数字、136 开头」），不能只写「必填」
6. **状态枚举**：数据字典中的状态字段必须列出所有枚举值
7. **可追溯**：所有"待补充"的地方在文档末尾的「TODO 清单」中汇总

### 禁止行为

1. **不要凭空编造**：原型 / 资料中没有的信息，写「待补充」，不要瞎填
2. **不要省略章节**：即使某章节暂时无内容，也保留标题，标注「本期不涉及」或「待补充」
3. **不要照搬术语**：要根据产品的真实业务场景翻译成具体的术语
4. **不要写过于宽泛的需求说明**：每个功能的「需求说明」必须包含「谁、在什么场景、做什么、达到什么目的」

## 三、文档输出规范

### 文件命名

```
PRD-[功能名]-V[版本号].md
```

例：
- `PRD-用户登录-V1.0.md`
- `PRD-借款申请-V2.1.md`

### 存放位置

```
[项目根目录]/02-方案/
```

### 多版本管理

- 修改但未评审 → V1.1、V1.2（小版本号）
- 评审通过 → 升级到 V2.0
- 文件内的「版本历史」表格必须同步更新

---

## 四、生成后检查清单（AI 自检）

生成完成后，AI 应自检以下要点并在对话中报告：

- [ ] 章节是否完整（1~8 全部存在）
- [ ] 是否有「待补充」字段未标注
- [ ] 每个功能的业务流程是否闭合
- [ ] 异常流程是否至少覆盖 1 种场景
- [ ] 数据字典是否包含所有界面字段
- [ ] Review Checklist 是否针对本需求做了初步勾选
- [ ] TODO 清单是否汇总在文档末尾

---

## 五、TODO 清单格式（生成时附加在文档末尾）

```markdown
## TODO 清单

> 本次生成中信息不足、需要产品经理补充的内容：

- [ ] 7.1.6 数据字典 - 用户表的「用户等级」字段枚举值待确认
- [ ] 7.2.5 异常流程 - 验证码超时的处理策略待补充
- [ ] 2.3 目标 - 量化指标待与运营方对齐
```

## 六、特殊场景处理

### 当原型中有动效或交互时

在「界面交互」章节增加「交互说明」子段：

```markdown
**交互说明**：
- 点击登录按钮后，按钮变 loading 态，2 秒内不可重复点击
- 验证码输入完成自动触发提交
```

### 当涉及多端时（App / 小程序 / 后台）

在功能模块标题后加端标识：

```markdown
### 7.1 用户登录（C 端 App）
### 7.2 用户管理（B 端后台）
```

### 当涉及 API 时

在「数据字典」后增加「接口说明」：

```markdown
#### 7.1.8 接口说明

| 接口名 | 路径 | 方法 | 入参 | 出参 |
| --- | --- | --- | --- | --- |
| 登录 | /api/login | POST | phone, code | token, userInfo |
```
