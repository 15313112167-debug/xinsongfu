# **三、健康风险数据
这是业务规则和合规的核心。

## **3.1 疾病史**

| **字段** | 紧急程度 |
| ------ | ---- |
| 高血压    | p0   |
| 糖尿病    | p0   |
| 高血脂    |      |
| 脂肪肝    |      |
| 痛风     |      |
| 甲状腺问题  |      |
| 心脑血管疾病 |      |
| 睡眠呼吸暂停 |      |
多选，支持“其他”x

举例：(生酮饮食)
肾脏负担：大量脂肪代谢、高蛋白加重肾脏滤过，肾病、肾功能异常人群严禁生酮；
血脂异常：饱和脂肪过量易升高坏胆固醇，高血脂、冠心病慎用；
酮症酸中毒风险：普通人节食生酮多为良性饥饿酮，糖尿病人自行生酮极易诱发糖尿病酮症酸中毒（急症）；
内分泌紊乱：女性易停经、脱发、月经紊乱；
便秘、胆结石高发：膳食纤维摄入不足、胆汁淤积。

# **十一、重点区分的“数据层级”**

这是产品架构里非常重要的一件事。

## **A. 用户输入数据**

例如：

- 体重
- 饮食
- 睡眠

特点：

- 不一定准确
- 允许修改

---

## **B. 设备同步数据**

例如：

- 体脂秤
- 手环

特点：

- 可信度更高
- 自动同步

---

## **C. 系统计算数据**

例如：

- BMI
- 风险等级
- 体重平台期（用户持续执行减重计划，但体重在一段时间内基本不下降。）

特点：

- 不允许用户修改

---

## **D. AI分析数据**

例如：

- AI评餐
- AI健康建议
- AI风险预测

特点：

- 需要标识“仅供参考”

---

# **十二、业务的关键问题**

## **1. 哪些数据是必须采集？**

因为：

- 注册流程长度
- 用户转化率  
    会受很大影响。

---

## **2. 哪些数据属于长期核心指标？**

因为：

- 决定首页设计
- 决定趋势图
- 决定运营重点

---

## **3. 哪些数据未来需要接设备？**

因为：

- 数据结构要提前设计
- 同步逻辑很复杂

---

## **4. 哪些数据会触发预警？**

这是规则引擎核心。

---

## **5. 哪些数据涉及医疗风险？**

比如：

- 血糖
- 尿酮
- 疾病史
- 用药

这会影响：

- 合规
- 隐私协议
- 风险免责声明
