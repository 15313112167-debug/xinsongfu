

# **一、增重业务本质（建议先和业务统一）**

## **增重 ≠ 纯长体重**

一定要先明确：

| **类型**        | **说明**     |
| --------------- | ------------ |
| 健康增重        | 增加健康体重 |
| 增肌            | 增加肌肉量   |
| 营养恢复        | 改善营养不良 |
| 术后恢复        | 恢复身体状态 |
| 儿童/青少年发育 | 特殊规则     |
| 老年营养改善    | 防肌少症     |

因为不同目标：

- 规则完全不同
- 饮食方案不同
- 风险点不同
- 数据指标不同

------

# **二、增重用户需要新增采集的数据**

------

# **1. 增重目标数据**

## **建议新增字段**

| **字段**     | **用途**         |
| ------------ | ---------------- |
| 增重目标类型 | 长体重/增肌/恢复 |
| 目标体重     | 核心             |
| 目标肌肉量   | 增肌核心         |
| 目标周期     | 计算节奏         |
| 理想体型     | 塑形方向         |
| 增重原因     | 偏瘦/健身/术后   |

------

# **2. 身体状态数据（增重更重要）**

减重更关注脂肪。

增重更关注：

- 肌肉
- 营养状态
- 蛋白
- 基础代谢

------

## **重点指标**

| **指标**      | **重要性** |
| ------------- | ---------- |
| 骨骼肌率      | 极高       |
| 肌肉量        | 极高       |
| 去脂体重      | 极高       |
| 基础代谢(BMR) | 极高       |
| 蛋白率        | 高         |
| 水分率        | 高         |
| BMI           | 中         |
| 体脂率        | 中         |

------

# **3. 饮食能力数据（增重特别关键）**

很多偏瘦用户的问题不是“不吃”。

而是：

- 吸收差
- 吃不下
- 饮食结构问题

------

## **建议新增字段**

| **字段**     | **用途**     |
| ------------ | ------------ |
| 食欲评分     | 判断摄入能力 |
| 每日餐次     | 是否少餐     |
| 是否经常跳餐 | 核心         |
| 胃口情况     | 食欲分析     |
| 消化情况     | 营养吸收     |
| 是否容易腹泻 | 风险         |
| 是否乳糖不耐 | 饮食方案     |
| 蛋白摄入量   | 增肌关键     |

------

# **4. 运动数据（增肌场景核心）**

减重：

- 关注消耗

增肌：

- 更关注力量训练

------

## **建议新增字段**

| **字段**     | **用途**     |
| ------------ | ------------ |
| 力量训练频率 | 核心         |
| 训练部位     | 训练计划     |
| 训练重量     | 增肌分析     |
| 训练组数     | 强度分析     |
| 有氧占比     | 防止过度消耗 |
| 恢复情况     | 防止掉肌     |
| 蛋白补充情况 | 增肌关键     |

------

# **5. 营养风险数据**

这是增重里容易忽略但非常重要的部分。

------

## **建议新增**

| **字段**     | **用途** |
| ------------ | -------- |
| 营养不良风险 | 风险等级 |
| 掉肌风险     | 重要     |
| 低蛋白风险   | 核心     |
| 长期低热量   | 风险     |
| 贫血情况     | 营养状态 |
| 维生素缺乏   | 营养评估 |

------

# **三、增重相关核心规则（重点沟通）**

------

# **1. 偏瘦规则**

建议业务明确：

## **BMI偏瘦阈值**

| **BMI** | **状态** |
| ------- | -------- |
| <18.5   | 偏瘦     |
| <17     | 明显偏瘦 |
| <16     | 营养风险 |

------

## **需要确认的问题**

| **问题**                         |
| -------------------------------- |
| 偏瘦用户是否自动切换“增重模式”？ |
| 是否禁止继续减重？               |
| 是否推荐营养师介入？             |
| 是否限制低热量方案？             |

------

# **2. 增重速度规则**

和减重一样，也要控制节奏。

------

## **建议沟通**

| **问题**          | **示例**     |
| ----------------- | ------------ |
| 每周最大增重      | 0.3~0.5kg？  |
| 是否允许快速增重  | 容易长脂肪   |
| 肌肉增长速度      | 是否单独计算 |
| 是否区分增脂/增肌 | 很重要       |

------

# **3. 增肌规则（非常关键）**

如果后面想做健身场景，这是核心。

------

## **需要明确**

| **问题**                     |
| ---------------------------- |
| 肌肉增长如何判断？           |
| 是否以肌肉量为核心指标？     |
| 是否允许体重不变但肌肉上涨？ |
| 是否关注掉肌？               |
| 是否做训练周期？             |

------

# **4. 热量盈余规则**

减重是热量缺口。

增重是热量盈余。

------

## **建议重点确认**

| **问题**              |
| --------------------- |
| 每日热量盈余目标      |
| 如何计算TDEE          |
| 是否动态调整热量      |
| 是否区分训练日/休息日 |
| 蛋白目标如何制定      |

------

# **5. 蛋白摄入规则**

增肌产品最核心规则之一。

------

## **建议确认**

| **问题**           |
| ------------------ |
| 每kg体重蛋白目标   |
| 是否限制最高蛋白   |
| 是否推荐蛋白粉     |
| 是否接营养补剂     |
| 是否需要营养师审核 |

这里涉及一定合规风险。

------

| 蛋白严重不足   | 是           |
| 训练过量       | 是           |
| 长期体重不增长 | 是           |

# **七、增重与减重共存时，你必须提前统一的系统设计**

这是产品架构里最容易踩坑的地方。

------

# **1. 目标体系统一**

不要只设计：

- 减重目标

而是：

| **类型** |
| -------- |
| 减脂     |
| 增重     |
| 增肌     |
| 维持     |
| 塑形     |

------

# **2. 推荐逻辑统一**

系统不要写死：

“体重下降 = 好”

因为增重用户是相反的。

------

# **3. 健康评分体系统一**

比如：

| **用户类型** | **目标**     |
| ------------ | ------------ |
| 减脂用户     | 降脂         |
| 增肌用户     | 长肌肉       |
| 偏瘦用户     | 增加健康体重 |

------

# **4. AI建议体系统一**

例如：

## **减重用户**

- 控制热量
- 增加消耗

------

## **增重用户**

- 增加热量
- 增加蛋白
- 提高恢复

------

# **八、重点追问业务的增重问题**

## **1. 产品到底是“减重产品”还是“体重管理产品”？**

这会决定整个产品架构。

------

## **2. 是否支持增肌场景？**

因为：

- 数据结构会完全不同
- 训练体系会完全不同

------

## **3. 是否支持青少年/老年营养改善？**

这会影响：

- 合规
- 医学规则
- 风险控制

------

## **4. 增重的核心指标是什么？**

是：

- 体重？
- 肌肉？
- BMI？
- 体脂？

一定要统一。

------

## **5. 增重是否需要营养师强介入？**

很多偏瘦用户其实有：

- 营养问题
- 疾病问题
- 心理问题

不能简单做“吃更多”。
